# Internationalization (i18n)

## Overview

The site supports English (`en`) and Chinese (`zh`) using [next-intl](https://next-intl-docs.vercel.app/).

## Configuration

### Locale Config (`i18n/config.ts`)

```typescript
export const locales = ['en', 'zh'] as const;
export const defaultLocale = 'en';
export type Locale = (typeof locales)[number];
```

- **Locale prefix:** `always` — every URL starts with `/en/` or `/zh/`
- **Default locale:** `en` (root `/` redirects to `/en`)

### Request Handler (`i18n/request.ts`)

Uses `getRequestConfig` from `next-intl/server` to load messages dynamically:

```typescript
const messages = (await import(`../messages/${locale}.json`)).default;
```

## URL Structure

```
/en/                    # English homepage
/zh/                    # Chinese homepage
/en/projects/           # English projects
/zh/projects/           # Chinese projects
/en/services/kitchen/   # English kitchen service
/zh/services/kitchen/   # Chinese kitchen service
```

## Translation Files

Located in `messages/`:

| File | Language | Keys |
|------|----------|------|
| `en.json` | English | Navigation, CTAs, section headings, form labels, meta, footer |
| `zh.json` | Chinese | Same keys, Chinese translations |

Key namespaces: `nav`, `cta`, `hero`, `section`, `label`, `footer`, `benefits`, `services`, `projects`, `blog`, `contact`, `stats`, `metadata`, `serviceBenefits`, `locationBenefits`, `areaBenefits`, `areas` (includes `faqTitle`, `aboutArea`)

### Usage in Components

```typescript
// Server component
import { useTranslations } from 'next-intl';

export default function MyComponent() {
  const t = useTranslations('Navigation');
  return <nav>{t('home')}</nav>;
}
```

### ICU MessageFormat

Some keys use ICU plural format for grammatically correct count displays:

```json
"projects": "{count, plural, one {project} other {projects}}"
```

Pass the `count` parameter when calling `t()`:

```typescript
{t('filter.projects', { count: items.length })}
```

Note: ICU plural without `#` outputs only the word (not the number). The component displays the count separately.

### Dynamic Placeholders in Meta Descriptions

Meta description keys use `{years}` placeholder for dynamic years of experience (computed from founding year). The `generateMetadata` functions fetch company data and pass it:

```typescript
const [t, company] = await Promise.all([
  getTranslations({ locale, namespace: 'metadata.home' }),
  getCompanyFromDb(),
]);
description: t('description', { years: company.yearsExperience })
```

## Bilingual Data Model

Content uses the `Localized<T>` type, whether stored in static files (`lib/data/`) or fetched from the database (`lib/db/queries.ts`):

```typescript
type Localized<T> = { en: T; zh: T };

// Example: project title
title: { en: 'Kitchen Renovation', zh: '厨房装修' }
```

Helper functions extract the right locale:

```typescript
function getLocalizedProject(project: Project, locale: Locale) {
  return {
    title: project.title[locale],
    description: project.description[locale],
    // ...
  };
}
```

## Database Columns

The DB schema uses `*En` / `*Zh` suffix convention:

```typescript
titleEn: varchar('title_en', { length: 256 }),
titleZh: varchar('title_zh', { length: 256 }),
```

## Navigation

`navigation.ts` exports locale-aware navigation utilities:

```typescript
import { Link, redirect, usePathname, useRouter } from '@/navigation';
```

These automatically include the current locale prefix in URLs.

## Locale Switching

The `Navbar` component includes a locale switcher that preserves the current path when switching between English and Chinese. The switcher uses `<Link href={pathname} locale={otherLocale}>` (a crawlable anchor) rather than a `<button>` with `router.replace()`, so search engine crawlers can discover the alternate-language version directly from the nav.

## SEO

- Sitemap includes hreflang alternates for both locales + `x-default`
- OG locale mapping: `en` → `en_US`, `zh` → `zh_CN`
- `buildAlternates()` in `lib/utils.ts` generates `<link rel="alternate">` tags
