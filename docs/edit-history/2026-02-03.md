## Session 1 — Docker Local Dev Environment + Documentation

### Summary

Added Docker Compose setup with PostgreSQL and MinIO for local development. Implemented dual database driver support (Neon for production, pg Pool for local). Created asset URL rewriting system so images load from local MinIO instead of production. Created initial project documentation.

### Changes

#### Docker & Infrastructure

- **Created `docker-compose.yml`** — PostgreSQL 16 Alpine (port 5432), MinIO (ports 9000/9001), `createbucket` init container that creates the `reno-stars` bucket with public-read access
- **Created `.dockerignore`** — Standard exclusions for node_modules, .next, .git, etc.
- **Created `.env.local.example`** — Pre-filled template for local dev environment variables

#### Database Driver

- **Modified `lib/db/index.ts`** — Replaced hardcoded Neon driver with conditional logic: URLs containing `neon.tech` use `@neondatabase/serverless`, otherwise use `pg` Pool. Keeps the lazy Proxy pattern for safe build-time imports.
- **Modified `lib/db/seed.ts`** — Removed duplicate `new-westminster` service area entry (was duplicated at display orders 8 and 9). Renumbered subsequent entries.

#### Asset Storage

- **Created `lib/storage.ts`** — `getAssetUrl(url)` helper rewrites `reno-stars.com/wp-content/uploads/...` to `localhost:9000/reno-stars/uploads/...` when `STORAGE_PROVIDER=minio`. Also exports `ASSET_ORIGIN` constant for CSP configuration.
- **Created `scripts/seed-storage.ts`** — Downloads 21 production assets and uploads to local MinIO. Idempotent via HeadObject check.
- **Modified `lib/data/index.ts`** — Wrapped 8 URLs (6 images, 1 video, 1 logo) with `getAssetUrl()`
- **Modified `lib/data/projects.ts`** — Wrapped all 13 `hero_image` and 39 `images[].src` URLs with `getAssetUrl()`

#### Configuration

- **Modified `next.config.ts`** — Added MinIO to `images.remotePatterns` (`http://localhost:9000/reno-stars/**`)
- **Modified `.env.example`** — Added `STORAGE_PROVIDER` documentation
- **Modified `package.json`** — Added `pg`, `@types/pg`, `@aws-sdk/client-s3` dependencies. Added scripts: `docker:up`, `docker:down`, `docker:reset`, `storage:seed`, `dev:services`

#### Documentation (New)

- **Created `CLAUDE.md`** — Project context for AI assistants
- **Created `docs/README.md`** — Documentation index and quick start
- **Created `docs/architecture.md`** — System architecture, directory layout, design patterns
- **Created `docs/database.md`** — Schema overview, commands, connection pattern
- **Created `docs/local-dev.md`** — Docker setup, MinIO, storage seeding, full reset guide

## Session 3 — Migrate 5 Static Data Types to Database

### Summary

Migrated Company Info, Social Links, Services, Testimonials, and About Sections from hardcoded static data (`lib/data/`) to live PostgreSQL database queries (`lib/db/queries.ts`). Moved Navbar and Footer rendering from individual page components into the locale layout (`app/[locale]/layout.tsx`) so they can receive DB-fetched data as props. All 14 page components updated to accept `company` as a prop instead of importing it statically.

### Architecture Changes

#### New Query Layer (`lib/db/queries.ts`)

Created 5 cached async query functions using React `cache()` for request-level deduplication:

- `getCompanyFromDb()` — Fetches singleton company info, computes `yearsExperience` from `foundingYear`, applies `getAssetUrl()` to logo
- `getSocialLinksFromDb()` — Fetches active social links ordered by `displayOrder`
- `getServicesFromDb()` — Fetches services ordered by `displayOrder`, applies `getAssetUrl()` to image URLs
- `getTestimonialsFromDb()` — Fetches featured testimonials
- `getAboutSectionsFromDb()` — Fetches about sections, replaces `{yearsExperience}` placeholder with computed value

All functions return the same TypeScript types (`Company`, `SocialLink[]`, `Service[]`, `Testimonial[]`, `AboutSections`) as the former static data exports.

#### Layout-Level Navbar/Footer

- **`app/[locale]/layout.tsx`** now fetches `company`, `socialLinks`, and `services` from DB via `Promise.all`, imports `areas` from static data (not migrated)
- Renders `<Navbar company={company} areas={areas} />` and `<Footer company={company} socialLinks={socialLinks} services={services} areas={areas} />`
- Renders `<LocalBusinessSchema company={company} socialLinks={socialLinks} />`
- All 14 page components no longer render `<Navbar />` or `<Footer />`

#### Static Data Cleanup

- **`lib/data/index.ts`** — Removed: `company`, `socialLinks`, `wechatId`, `testimonials`, `aboutSections`, and their localization helpers (`getAllTestimonialsLocalized`, `getLocalizedTestimonial`, `getAboutSectionsLocalized`)
- **`lib/data/services.ts`** — Removed: `services` array, `getServices()`, `getServiceBySlug()`, `getAllServicesLocalized()`. Kept: `getLocalizedService()`, `serviceTypeToCategory`, `categoryToServiceType`
- `showroom` now uses literal values instead of referencing `company.address/phone/email`

### Schema Changes

- **`lib/db/schema.ts`** — Extended `socialPlatformEnum` with `'xiaohongshu'`, `'wechat'`, `'whatsapp'`. Added `foundingYear` (integer), `reviewCount` (integer), `geoLatitude` (varchar), `geoLongitude` (varchar) columns to `companyInfo` table.

### Seed Updates (`lib/db/seed.ts`)

- Services: corrected icon names and descriptions to match original static data, added `imageUrl`
- Social links: 5 platforms (xiaohongshu, wechat, instagram, facebook, whatsapp) with real URLs
- Company info: added `logoUrl`, `foundingYear: 1997`, `reviewCount: 150`, geo coordinates
- About sections: seeded with bilingual content, `{yearsExperience}` placeholder in `ourJourney`
- Testimonials: 3 featured entries (Sarah M., David L., Jennifer K.)

### Component Changes

#### Shared Components (accept props instead of importing static data)

- **`Navbar.tsx`** — Props: `{ company: Company; areas: ServiceArea[] }`
- **`Footer.tsx`** — Props: `{ company: Company; socialLinks: SocialLink[]; services: Service[]; areas: ServiceArea[] }`. Services localized client-side via `useLocale()` + `useMemo`
- **`CTASection.tsx`** — Added `phone?: string` prop, removed static `company` import
- **`LocalBusinessSchema.tsx`** — Props: `{ company: Company; socialLinks: SocialLink[] }`
- **`ServiceSchema.tsx`** — Props: `{ company: Company; ... }`
- **`ArticleSchema.tsx`** — Props: `{ company: Company; ... }`

#### Page Components (14 files in `components/pages/`)

All page components:
- Removed `<Navbar />` and `<Footer />` from JSX and imports
- Removed `company` import from `@/lib/data`
- Added `company: Company` to props (except ThankYouPage)
- Pass `phone={company.phone}` to CTASection

Special cases:
- **HomePage** — Also receives `services`, `testimonials`, `aboutSections` as props
- **ServicesPage** — Also receives `services` as prop
- **ServiceDetailPage** — Also receives `service: Service` as prop (instead of calling removed `getServiceBySlug`)
- **ServiceLocationPage** — Also receives `service: Service` as prop
- **AreaPage** — Also receives `services: LocalizedService[]` as prop (instead of calling removed `getAllServicesLocalized`)

#### Server Pages (~14 files in `app/[locale]/**/page.tsx`)

All server pages:
- Import query functions from `@/lib/db/queries`
- Fetch data with `await getCompanyFromDb()` (often via `Promise.all`)
- Pass data as props to client components

Service server pages:
- `generateStaticParams` uses `serviceTypeToCategory` keys (static, no DB needed at build time) instead of `getServicesFromDb()`
- `generateMetadata` and `Page` use `getServicesFromDb()` to find service by slug

#### Sitemap (`app/sitemap.ts`)

- Changed from async (DB query) to sync using `serviceTypeToCategory` keys for service slugs
- No longer requires DB connection at build time

### Build Considerations

- `DATABASE_URL` is now required at build time because `layout.tsx` and all pages fetch data from DB during pre-rendering
- `generateStaticParams` and `sitemap.ts` use static `serviceTypeToCategory` keys to avoid DB dependency during page data collection phase
- The lazy DB proxy pattern still prevents crashes at import time
