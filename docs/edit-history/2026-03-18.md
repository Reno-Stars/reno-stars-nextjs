## Session 2: AI Output Review — Hybrid Validation for Project Descriptions

Added a two-layer validation system for AI-generated project descriptions to catch invalid scopes, bad service types, and slug-title mismatches.

### New Features

#### 1. Programmatic validation (`validateProjectDescription`)
- **Files:** `lib/ai/content-optimizer.ts`
- Pure function that runs immediately after Zod parsing in `optimizeProjectDescription()`
- **Scope enforcement:** Filters `selectedScopes` to only entries existing in the provided scope list; falls back to all scopes if every AI-selected scope was invalid
- **ServiceType enforcement:** Sets `serviceType` to `null` if not in the allowed list
- **Slug-title consistency:** Regenerates slug from title via `formatSlug()` when slug and title share fewer than `MIN_SLUG_TITLE_OVERLAP` (2) significant words (stop words excluded)
- Returns `{ corrected, corrections }` — corrections array used for logging
- Deep-copies `selectedScopes` array to avoid mutating input

#### 2. Lightweight AI review (`reviewProjectDescription`)
- **Files:** `lib/ai/content-optimizer.ts`
- Second, cheaper AI call using `gpt-4o-mini` with `temperature: 0.1` and 512 max tokens
- Sends only the raw notes + key generated fields (title, serviceType, slug, scopes) to a focused reviewer prompt
- Returns only fields that need correction as a partial object; returns `null` if everything is correct
- Non-fatal — wrapped in try/catch, failures just log a warning and skip the review
- `ReviewCorrectionsSchema` uses strict Zod (no `.passthrough()`) to drop unknown fields

#### 3. Integration into `optimizeProjectDescription()`
- **Files:** `lib/ai/content-optimizer.ts`
- After Zod parse: (1) programmatic validation always runs, (2) AI review runs and applies corrections on top
- Review corrections are re-validated (serviceType checked against allowed list, slug sanitized through `formatSlug()`, scopes re-validated)
- Return type extended to `ProjectDescription & { corrections?: string[] }`
- All corrections logged via `console.log('[AI Validation]', ...)`

#### 4. Batch processor logging
- **Files:** `lib/batch/batch-processor.ts`
- `generateProjectMetadata()` logs corrections from the validation/review step
- Existing scope filter at save time (lines 1098-1103) remains as a safety net

#### 5. `maxTokensReview` config
- **Files:** `lib/ai/openai.ts`
- Added `maxTokensReview: 512` to `AI_CONFIG`

### Code Review Fixes

#### 6. ServiceType re-validation after AI review
- AI review could return a serviceType not in the allowed list — now re-validated after applying review corrections

#### 7. Slug sanitization after AI review
- Reviewer-provided slug is run through `formatSlug()` to ensure valid URL format

#### 8. Removed `.passthrough()` from ReviewCorrectionsSchema
- Unknown fields from AI response are now stripped by Zod

#### 9. Deep copy for selectedScopes in validateProjectDescription
- Changed `{ ...result }` to `{ ...result, selectedScopes: [...result.selectedScopes] }` to prevent mutation

#### 10. Set-based slug word comparison
- Slug words now use `Set.has()` instead of `Array.includes()`

#### 11. Named constant for slug overlap threshold
- Extracted magic number `2` to `MIN_SLUG_TITLE_OVERLAP`

#### 12. Single-pass scope partition
- Replaced two `.filter()` calls with one loop

#### 13. JSDoc placement fix
- Moved `type ReviewCorrections` above JSDoc block so doc comment associates with the function

### Tests

#### 14. Unit tests for `validateProjectDescription`
- **Files:** `tests/unit/lib/ai/content-optimizer.test.ts`
- 14 tests across 5 describe blocks: serviceType enforcement (4), scope enforcement (5), slug-title consistency (3), input immutability (1), combined corrections (1)

## Session 3: Fix Batch Upload AI — Zod Schema Resilience & Error Logging

Batch upload was silently producing fallback "Kitchen Renovation" data for ZIPs with valid `notes.txt`. Root cause: GPT-4o-mini sometimes omits SEO fields (`metaTitleEn`, `metaDescriptionEn`, etc.) from the JSON response, causing `ProjectDescriptionSchema.parse()` to throw a `ZodError`. The catch block in `generateProjectMetadata()` silently swallowed the error and returned `null`, triggering kitchen-specific fallback data.

### Bug Fix

#### 1. Added `.default('')` to optional Zod schema fields
- **Files:** `lib/ai/content-optimizer.ts`
- `ProjectDescriptionSchema`: 16 fields changed from `z.string()` to `z.string().default('')` — `locationCity`, `poNumber`, `budgetRange`, `durationEn/Zh`, `challengeEn/Zh`, `solutionEn/Zh`, `badgeEn/Zh`, `excerptEn/Zh`, `metaTitleEn/Zh`, `metaDescriptionEn/Zh`, `focusKeywordEn/Zh`, `seoKeywordsEn/Zh`
- `SiteDescriptionSchema`: same pattern for 14 fields
- Core fields (`slug`, `titleEn/Zh`, `descriptionEn/Zh`, `detectedLanguage`) remain required — these are essential and their absence indicates a real AI failure
- Prevents ZodError when the AI omits non-critical fields

#### 2. Error logging in AI metadata generators
- **Files:** `lib/batch/batch-processor.ts`
- `generateSiteMetadata()`: catch block now logs `console.error` with folder name and error message (was completely silent)
- `generateProjectMetadata()`: catch block now logs `console.error` with folder name and error message (was completely silent)
- Corrections logging changed from `console.log` to `console.warn` (matches eslint allowed list)
