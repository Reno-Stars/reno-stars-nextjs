   - Removed unused `Fragment` import.

3. **Contact page heading semantics** (`ContactPage.tsx`)
   - "Request Free Consultation" was an `<h2>` with only a single `<p>` underneath — thin content under a prominent heading from a crawler's perspective.
   - Changed from `<h2>` to `<p>` with same visual styling (`text-lg font-bold`). It's a callout, not a section heading.

4. **Rating display fix** (`HeroSection.tsx`, `Footer.tsx`, `ContactPage.tsx`, `ProcessPage.tsx`, `HomePage.tsx`, `layout.tsx`, `contact/page.tsx`, `process/page.tsx`)

**Session 10** — Fix batch upload 413 error on Vercel production. Batch ZIP upload failed with "413 Request Entity Too Large" because 10 MB chunks were sent through the Vercel API route, which has a 4.5 MB body size limit. S3 multipart requires ≥5 MB per non-final part, so chunks can't be shrunk below 4.5 MB.

### Problem

1. **413 error**: Vercel serverless functions reject request bodies >4.5 MB. The old flow proxied 10 MB chunks through the API route to S3.
2. **JSON parse crash**: The client called `partRes.json()` on the 413 plain-text error response ("Request Entity Too Large"), causing an unhandled exception.
3. **CORS ETag issue**: After switching to presigned URLs (direct browser→S3 upload), browsers couldn't read the `ETag` response header from cross-origin S3 responses because the bucket's CORS config doesn't include `Access-Control-Expose-Headers: ETag`.

### Solution

Switched to presigned URL pattern (already used for single image uploads):

1. **PUT handler** returns a presigned URL instead of proxying chunk data. Client sends only query params (tiny request, no body).
2. **Client** uploads chunks directly to S3 via presigned URLs, bypassing Vercel entirely.
3. **PATCH handler** retrieves ETags server-side via `ListPartsCommand` instead of relying on client-provided ETags. Client sends `{ uploadId, totalParts }` and the server verifies all parts exist on S3.

### Changes Made

1. **Upload route** (`batch-upload/api/[jobId]/upload/route.ts`)
   - Added `getSignedUrl` import from `@aws-sdk/s3-request-presigner`
   - Added `ListPartsCommand` import
   - PUT: Accepts `contentLength` query param, returns `{ presignedUrl }` (no body read)
   - PATCH: Accepts `{ uploadId, totalParts }` instead of `{ uploadId, parts: [...] }`. Calls `ListPartsCommand` to get server-side ETags, verifies part count matches, then completes.

2. **Client** (`batch-upload/BatchUploadClient.tsx`)
   - Each chunk: (A) fetch presigned URL from API → (B) PUT chunk directly to S3
   - Removed ETag tracking — server handles it
   - S3 errors shown as generic message (no XML leak)

### Files Modified

**Session 11** — Fix homepage marquee duplicate content in page output. Testimonials and partner logos appeared multiple times in the server-rendered HTML because the CSS marquee implementation cloned items server-side for the seamless loop animation. Text parsers, crawlers, and QA tools saw the duplicate content blocks.

### Problem

The original marquee implementation rendered all items twice in the SSR output (once crawlable, once `aria-hidden`). After adding `repeatCount` expansion to fill wide viewports, the duplicates multiplied further (4× with 5 reviews). `aria-hidden` only hides from assistive tech — the content was still present in the HTML for parsers and extractors.

### Solution

Created a reusable `Marquee` client component that renders clones **only after hydration**, so the server-rendered HTML contains each item exactly once.

### Changes Made

1. **New file**: `components/home/Marquee.tsx` — Client component (`'use client'`) that accepts `children` (semantic, always rendered) and `clones` (decorative, client-only via `useState`/`useEffect`). Animation starts only after clones mount (inline `style` with `marquee-scroll` keyframes). Respects `prefers-reduced-motion` with live `matchMedia` listener. Hover/focus pause via `useCallback` event handlers.

2. **Global keyframes**: `app/globals.css` — Added `@keyframes marquee-scroll` (shared by both marquee instances, replacing two inline `<style>` blocks).

3. **TestimonialsSection** (`components/home/TestimonialsSection.tsx`) — Replaced inline `<style>` + manual two-pass render with `<Marquee>`. Calculates `repeatCount` based on `CARD_WIDTH=340` / `MIN_TRACK_WIDTH=2000`. Passes original reviews as `children`, `repeatCount * 2 - 1` extra sets as `clones`. Also extracted `MS_PER_DAY` constant in `getRelativeTime()`.

4. **PartnersSection** (`components/home/PartnersSection.tsx`) — Same refactor using `<Marquee>`. Calculates `repeatCount` based on `LOGO_WIDTH=192` / `MIN_TRACK_WIDTH=2000`. Division-by-zero guard for empty `visiblePartners`.

### SSR Output (before → after)

| | Before | After |
|---|--------|-------|
| Reviews in HTML | 10–20 (5 reviews × repeatCount × 2) | 5 (each review once) |
| Partner logos in HTML | 14–28 | 7 visible + sr-only hidden list |
| `<style>` blocks | 2 inline (one per section) | 0 (shared keyframes in globals.css) |
| Animation start | Immediate (broken before hydration) | After clones mount (no flash) |
