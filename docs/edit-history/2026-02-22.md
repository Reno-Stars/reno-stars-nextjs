## 1. DataTable Drag-and-Drop Reordering

### Problem

Services, service areas, FAQs, and trust badges admin lists had `displayOrder` columns but no drag-and-drop reordering. Gallery and Partners already supported drag reorder via `useDragReorder` hook on visual card grids, but table-based lists had no reorder capability.

### Solution

Enhanced `DataTable` with an optional `dragReorder` prop. When provided:
- Drag handle column prepended to each row
- Column sorting auto-disabled (reorder requires stable order)
- Dragging disabled while search is active (handles grayed out)
- Gold insertion line indicator on the drop target row (top or bottom edge depending on drag direction)
- Dragged row shown at 40% opacity

### DataTable Changes (`components/admin/DataTable.tsx`)

- Exported new `DragReorderConfig` interface (subset of `useDragReorder` return values)
- Added `dragReorder?: DragReorderConfig` to `DataTableProps`
- `dragIndexMap` memo: builds a `Map<rowKey, index>` for the current page, recomputed only when `draggedId`/`dragOverId` change
- `getInsertionEdge(rowKey)` function: returns `'top'` | `'bottom'` | `null` — when dragging down (D < T), shows gold line on bottom of target; when dragging up (D > T), shows top
- `isDragEnabled` computed from `!!dragReorder && !isSearching`
- `effectiveSortable` per column: `col.sortable && !dragReorder` — suppresses sort arrows when drag is active
- `totalCols` includes the optional drag handle column for correct `colSpan` calculations
- Row styles: `borderBottom: 2px solid GOLD` or `borderTop: 2px solid GOLD` for insertion line, subtle `${GOLD}08` background tint on target, `opacity: 0.4` on dragged row, `cursor: grab` when drag enabled
- Imported `DragHandleIcon` component for the handle cells

### Server Actions (4 new)

All follow the existing `reorderGalleryItems` pattern: validate UUIDs, `Promise.all` update `displayOrder: i`, revalidate admin + public paths.

| File | Function | Table | `updatedAt`? |

### List Client Changes (4 files)

Each list client:
- Imports `useDragReorder` hook + corresponding reorder server action
- Calls hook with entity-specific `isIncluded`:
  - Services: `() => true` (all items visible, no active/inactive toggle)

- Removed `sortable: true` from column definitions (incompatible with drag mode)

## 2. Upload Cache-Busting Fix

### Problem

When uploading to `ImageUrlInput` or `ImagePairEditor` with a `slug` prop, the `customKey` was deterministic (e.g., `{slug}-icon`). Re-uploading the same file type overwrote the S3 object but returned the identical URL. The browser served the cached old image, making it look like the upload and subsequent save had no effect.

### Solution

Appended a base-36 timestamp suffix to the `customKey` so each upload produces a unique S3 key and URL:

| Component | Before | After |
|-----------|--------|-------|
| `ImageUrlInput` | `{slug}-{imageRole}` | `{slug}-{imageRole}-{ts}` |
| `ImagePairEditor` (before) | `{slug}-before-renovation-{idx}` | `{slug}-before-renovation-{idx}-{ts}` |
| `ImagePairEditor` (after) | `{slug}-after-renovation-{idx}` | `{slug}-after-renovation-{idx}-{ts}` |

Where `ts = Date.now().toString(36)` (e.g., `m3x7k2f`).

### Cleanup

The initial fix also added an `uploadVersion` state counter with `?v=N` cache-buster query parameter on the preview image. This was subsequently removed as dead code — with unique URLs, the browser naturally fetches the new image without cache-busting params.
