## Presigned S3 URL Upload (Batch Upload)

### Problem

The Next.js proxy limits request bodies to 10MB by default. Real ZIP files for batch upload exceeded this limit, causing `Failed to parse body as FormData` errors even though the app's own `MAX_ZIP_SIZE` is 100MB.

### Solution: Direct-to-S3 upload via presigned URL

Instead of routing the ZIP through the proxy, the client now uploads directly to S3 using a presigned PUT URL.

**New upload flow:**

1. Client sends JSON metadata (`fileName`, `fileSize`, `generateBlog`) to `POST /admin/batch-upload/api`
2. Server validates input, checks S3 availability, creates `batch_upload_jobs` row, generates a presigned PUT URL (10-minute expiry) for `temp/batch/{jobId}.zip`
3. Client uploads ZIP directly to S3 via `PUT` to the presigned URL (bypasses proxy)
4. Client triggers processing via `POST /admin/batch-upload/api/{jobId}/process`
5. Batch processor downloads ZIP from S3, processes it, then deletes the temp file (unchanged)

### CORS Requirement

The presigned URL approach requires CORS to be configured on the S3-compatible storage provider (MinIO, Cloudflare R2, AWS S3) to allow PUT requests from the app's origin. Without CORS configuration, the browser's direct S3 upload will fail with a network error.
