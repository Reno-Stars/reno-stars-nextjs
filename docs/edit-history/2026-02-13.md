## Project Modal Enhancements (`components/ProjectModal.tsx`)

### Mobile Swipe Support

Added touch event handling for mobile image navigation:

```typescript
// Minimum swipe distance to trigger navigation (in pixels)
const SWIPE_THRESHOLD = 50;

const handleTouchStart = useCallback((e: React.TouchEvent) => {
  touchStartRef.current = {
    x: e.touches[0].clientX,
    y: e.touches[0].clientY,
  };
}, []);

const handleTouchEnd = useCallback((e: React.TouchEvent) => {
  // Only trigger swipe if horizontal movement > vertical (avoid scroll conflicts)
  if (Math.abs(deltaX) > Math.abs(deltaY) && Math.abs(deltaX) > SWIPE_THRESHOLD) {
    if (deltaX < 0) handleNextImage();  // Swipe left → next
    else handlePrevImage();              // Swipe right → previous
  }
}, [handleNextImage, handlePrevImage]);
```

Touch overlay with z-index layering:
- Touch overlay at `z-[5]` captures swipe events
- Before/after badge at `z-10`
- Navigation arrows at `z-20`

## Admin Action Safety Improvements

### Insert-Before-Delete Pattern

Admin CRUD actions that update related records now use safer insert-before-delete pattern instead of transactions. The Neon HTTP driver does not support interactive transactions.

**Pattern:**
1. Fetch existing record IDs before modification
2. Insert new records first (old data remains as fallback if insert fails)
3. Delete old records by ID after successful insert

**`app/actions/admin/projects.ts`:**

```typescript
// Fetch existing related record IDs before modification
const [existingPairs, existingScopes, existingEps] = await Promise.all([
  db.select({ id: projectImagePairs.id }).from(projectImagePairs).where(eq(projectImagePairs.projectId, id)),
  db.select({ id: projectScopes.id }).from(projectScopes).where(eq(projectScopes.projectId, id)),
  db.select({ id: projectExternalProducts.id }).from(projectExternalProducts).where(eq(projectExternalProducts.projectId, id)),
]);

// Update project
await db.update(projects).set({ ...data, serviceId: svcRows[0].id }).where(eq(projects.id, id));

// Insert new related data first (old data still exists as fallback if insert fails)
if (pairData.length > 0) {
  await db.insert(projectImagePairs).values(pairData.map((pair) => ({ ...pair, projectId: id })));
}
// ... similar for scopes and external products

// Delete old related data (new data already safely inserted)
if (oldPairIds.length > 0) {
  await db.delete(projectImagePairs).where(inArray(projectImagePairs.id, oldPairIds));
}
// ... similar for scopes and external products
```

**Also applied to:**
- `app/actions/admin/sites.ts` — Same pattern for site image pairs

## Projects Page Helper (`components/pages/ProjectsPage.tsx`)

### `collectImagesFromPairs()` Helper

Extracted duplicate image collection logic into reusable helper:

```typescript
interface GalleryImage {
  src: string;
  alt: string;
  is_before?: boolean;
}

/**
 * Collects unique images from image pairs, skipping duplicates.
 * Used by both site and project conversion to avoid code duplication.
 */
function collectImagesFromPairs(
  pairs: Array<{
    afterImage?: { src: string; alt: string } | null;
    beforeImage?: { src: string; alt: string } | null;
  }> | undefined,
  existingImages: GalleryImage[],
  fallbackAlt: string
): void {
  if (!pairs) return;
  for (const pair of pairs) {
    if (pair.afterImage && !existingImages.some((img) => img.src === pair.afterImage!.src)) {
      existingImages.push({ src: pair.afterImage.src, alt: pair.afterImage.alt || fallbackAlt, is_before: false });
    }
    if (pair.beforeImage && !existingImages.some((img) => img.src === pair.beforeImage!.src)) {
      existingImages.push({ src: pair.beforeImage.src, alt: pair.beforeImage.alt || fallbackAlt, is_before: true });
    }
  }
}
```

Used by both `sitesAsDisplayProjects` and `projectsAsDisplay` memoized conversions.

## Session 2: AI Blog Generation

### New Files

- **`lib/ai/blog-generator.ts`** — AI generation logic using GPT-4o
- **`app/actions/admin/generate-blog.ts`** — Server actions `generateBlogFromProject()` and `generateBlogFromSite()`

### How It Works

1. Admin clicks "Generate Blog Post" button on the site detail page
2. Server action fetches project/site data with all relations (image pairs, scopes, products)
3. Data is formatted into a context string and sent to GPT-4o with a structured prompt
4. AI generates bilingual blog post with HTML content, SEO metadata, and slug
5. Response is validated with Zod schema (`BlogGenerationSchema`)
6. Draft blog post is inserted into the database (unpublished)
7. Admin is redirected to the blog editor to review and publish

### Safety Features

- SEO fields are truncated to match DB column max lengths (`SEO_META_TITLE_MAX`, etc.)
- AI-generated slugs are sanitized (lowercase, hyphens only, no consecutive hyphens)
- Slugs are deduplicated via `ensureUniqueSlug()`
- Truncated response detection (`finish_reason === 'length'`) throws clear error
- Auth required (`requireAuth()`) on both actions

### Admin UI

`SiteDetailClient.tsx` adds a "Generate Blog Post" button in the detail panel header. Works for both individual projects and the site as a whole. Uses `useTransition` for non-blocking UI during generation.

## Session 3: AI Blog Generation Code Review Fixes

**Session 3** focused on **code review and hardening** of the AI blog generation feature from Session 2. A comprehensive 13-issue code review was performed, and all issues were resolved.

### Issues Fixed

#### Critical (2)

**#1 — SEO field truncation**: AI-generated SEO fields (metaTitle, metaDescription, focusKeyword) could exceed DB column max lengths, causing insert failures.

- Added `truncateField(value, max)` helper in `generate-blog.ts`
- Imports `SEO_META_TITLE_MAX` (70), `SEO_META_DESCRIPTION_MAX` (155), `SEO_FOCUS_KEYWORD_MAX` (50) from schema
- All SEO fields truncated before DB insert
- Added `focusKeyword under 50 characters` instruction to the AI prompt

**#2 — Token limit insufficient**: `maxTokensContent` (8192) was too low for bilingual blog generation (800-1200 words x2 languages + SEO fields in JSON).

- Added `maxTokensBlogGeneration: 16384` to `AI_CONFIG` in `openai.ts`
- Both `generateBlogFromProjectData` and `generateBlogFromSiteData` now use `AI_CONFIG.maxTokensBlogGeneration`

#### Warnings (3)

**#3 — Slug sanitization**: AI-generated slugs could contain characters rejected by `isValidSlug()` (which rejects consecutive hyphens like `a--b`).

- Added `sanitizeSlug(raw)` function: lowercase, replace invalid chars with hyphens, collapse consecutive hyphens, trim leading/trailing hyphens, fallback to `'blog-post'`
- Applied before `ensureUniqueSlug()` in `insertBlogDraft()`

**#4 — Duplicate insert logic**: Both `generateBlogFromProject` and `generateBlogFromSite` had near-identical blog post insert code.

- Extracted `insertBlogDraft(generated, featuredImageUrl, projectId)` shared async helper
- Handles slug sanitization, uniqueness, SEO truncation, and DB insert in one place

**#5 — Button shown for empty sites**: "Generate Blog Post" button was visible when site was selected but had no child projects, which would trigger a server error.

- Button now hidden when `selected === 'site' && projects.length === 0`

#### Suggestions (5)

**#6 — Button disabled during isPending**: Generate button was only disabled during `isBlogPending`, not during other transitions (`isPending` from delete/reorder).

- Button now disabled when `isBlogPending || isPending`
- Cursor and opacity also reflect combined state

**#7 — filter(Boolean) in site context**: Used `filter(Boolean)` on context lines array which could filter out intentional empty strings or falsy values.

- Rewrote site context building with explicit conditional `if` checks and `push()` calls

**#8 — Non-exported interfaces**: `ProjectDataForBlog` and `SiteDataForBlog` were not exported, preventing external use.

- Changed from `interface` to `export interface`

**#10 — Hardcoded feedback banner colors**: Blog message banner used hardcoded hex colors (`#e8f5e9`, `#2e7d32`, etc.) instead of theme tokens.

- Imported `SUCCESS`, `SUCCESS_BG`, `ERROR`, `ERROR_BG` from `lib/theme.ts`
- Feedback banner now uses theme tokens consistently

**#11 — Missing aria-busy**: Generate button lacked `aria-busy` attribute for screen reader feedback during loading.

- Added `aria-busy={isBlogPending}` to the button element

**#13 — Duration null check**: `buildProjectContext()` could produce `Duration: 2 weeks / null` when only EN duration existed.

- Now checks both `durationEn` and `durationZh` before building the combined line
- Falls back to EN-only line if ZH is null

#### Kept As-Is (2)

**#9 — Hardcoded author**: Author is hardcoded as `'Reno Stars'` in `insertBlogDraft()`. Kept as-is since this is a single-author company site.

**#12 — Rate limiting**: No rate limit on blog generation actions. Kept as low-risk since actions are admin-only behind `requireAuth()`.
