# Edit History: 2026-03-05

## Summary

Fixed three bugs in the batch upload ZIP parser and processor that prevented site-level image pairs from working correctly. Also hardened orphan cleanup logic found during code review.

---

## Batch Upload Bug Fixes

### Bug 1: Space separator in filenames (`lib/batch/zip-parser.ts`)

Before/after image filename matching used regexes that only accepted hyphens as separators (`before-1.jpg`). Real-world ZIPs from macOS (e.g., `Before 1.jpg`, `After 2.jpg`) use spaces, causing all before/after images to be treated as unpaired standalone images.

**Fix**: Changed regex from `/^before(?:-(\d+))?$/` to `/^before(?:[- ](\d+))?$/` (and same for `after`) to accept both hyphens and spaces.

### Bug 2: Site-level image pairs not stored (`lib/batch/types.ts`, `lib/batch/zip-parser.ts`)

`ParsedSite` type had no `imagePairs` field. When a ZIP had root-level images alongside subfolders (e.g., `8171/Before 1.jpg` next to `8171/Kitchen/`), the root images were incorrectly wrapped into a fake project instead of being stored as site-level image pairs in the `site_image_pairs` table.

**Fix**:
- Added `imagePairs: ImagePairEntry[]` to `ParsedSite` type
- Parser now routes root-level before/after/standalone images to `site.imagePairs` when subfolders exist (nested layout)
- For flat/single-folder layouts (no subfolders), `site.imagePairs` is `[]` — pairs belong to the project only, preventing duplicate DB rows

### Bug 3: Batch processor missing `site_image_pairs` insertion (`lib/batch/batch-processor.ts`)

The batch processor only inserted into `project_image_pairs`, never `site_image_pairs`. Root-level site images were silently lost.

**Fix**:
- Imported `siteImagePairs` from schema
- Added site-level image collection for S3 upload step
- Added site-level images to alt text AI generation step
- Inserted site image pairs into `site_image_pairs` table after site creation, before project creation

## Code Review Fixes

Code review found 3 critical issues in the initial fix, all resolved:

### Fix 1: Duplicate image pairs in flat/no-subfolder layouts

In flat layout (all images at root) and no-subfolder layout (single top-level folder), the same `rootPairs` array was assigned to both `site.imagePairs` and the child project's `imagePairs`. The batch processor would insert identical rows into both `site_image_pairs` and `project_image_pairs`.

**Fix**: For flat/no-subfolder layouts, set `site.imagePairs = []`. Site-level pairs are only populated for nested layouts where root images are distinct from project subfolder images.

### Fix 2: `cleanupOrphanedSite` ignores site image pairs

The orphan cleanup function only checked for child projects before deleting a site. If a failure occurred after site image pairs were inserted but before projects, the function deleted the site and cascade-deleted valid image pair rows.

**Fix**: `cleanupOrphanedSite()` now also checks `site_image_pairs` for the site before deleting. Returns early if either child projects or child image pairs exist.

### Fix 3: Orphan check used parsed count instead of actual insert count

The orphan check used `parsedSite.imagePairs.length` (parsed count) instead of tracking actual successful DB inserts. If all site image pair inserts failed, the site would remain orphaned.

**Fix**: Added `siteImagePairsCreated` counter incremented on successful insert. Orphan check now uses `siteImagePairsCreated === 0` instead of `parsedSite.imagePairs.length === 0`.
